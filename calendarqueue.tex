\section{Calendar Queue}
Ein Optimierungsansatz für das \ac{PES}-Problem ist die \ac{CQ}, 1988 vorgeschlagen von Randy Brown. In Anlehnung an einen Kalender, in den Termine eingetragen werden, gibt es in einer \ac{CQ} Tage (Buckets), die mehrere Events enthalten können, sowie Jahre, die aus einer Anzahl von Tagen bestehen. Die Buckets sind jeweils Zeiger auf eine (beliebig implementierte) Liste, in der die Events liegen. Zum \textit{enqueuen} muss der Bucket, in den das Event gehört durch die Formel
\begin{equation}
	bucket\_idx=\floor{\frac{t(e)}{bw}} mod M
\end{equation}
ermittelt werden, wobei $bucket\_idx$ der Zielbucket, $t(e)$ die Zeit des Events, $bw$ die aktuelle \textit{\aclu{bw}} (Länge des Tages) und $M$ die Anzahl der Buckets im Jahr ist. Anschließend wird das Element in den ermittelten Bucket eingefügt. Um zu \textit{dequeuen}, muss das kleinste Element aus dem aktuellen Bucket entfernt werden, wobei leere Buckets übersprungen werden.
Um auch Events \textit{enqueuen} zu können, die mehr als ein Jahr (Gesamtanzahl an Buckets) in der Zukunft liegen, wird das Datum im Event selbst gespeichert, sodass dieses bei einer \textit{dequeue}-Operation in den vorigen Jahren übersprungen werden kann. \cite{brown_cq}

\subsection{Optimierungen}
Die Länge eines Jahres und die Länge eines Tages werden dynamisch an die Anzahl der Events angepasst. Steigt die Zahl der Events auf mehr als das doppelte der Anzahl der Buckets, werden die Events in eine \ac{CQ} doppelter Größe kopiert, sodass durchschnittlich jedes Event in einem eigenen Bucket liegt. Analog halbiert sich die Größe der \ac{CQ} bei halb so viel Events wie Buckets in der \ac{CQ} vorhanden sind. Die \textit{\aclu{bw}} wird beim Vergrößern/Verkleinern angepasst. Sie sollte ungefähr so groß sein, wie die durchschnittliche Zeit zwischen zwei benachbarten Events.
Eine von Brown vorgeschlagene Heuristik, eine akzeptable \textit{\acl{bw}} zu ermitteln, ist, die ersten \textit{n} Elemente zu \textit{dequeuen}, die durchschnittliche Zeit zwischen diesen Events zu errechnen, und diese anschließend wieder zu \textit{enqueuen}. In Bezug auf statistische Aussagekraft muss \textit{n} hinreichend groß gewählt werden. \cite{brown_cq}
\subsection{Performance}
Die Calendar Queue hat eine theoretische Komplexität von $ O(1) $ für sowohl die \textit{dequeue}- als auch die \textit{enqueue}-Operation. 
Die amortisierte Zeit wird jedoch maßgeblich durch die Zeitverteilung der Events beeinflusst. Während die Performance bei absoluter Gleichverteilung optimal ist, ist bei großer Varianz des Abstandes der Events zueinander keine konsistente Performance gegeben. Wenn sich Events innerhalb eines Buckets häufen, dauert das \textit{enqueuen} länger, da die \ac{LL} länger wird. Wenn viel Zeit zwischen den Events liegt, müssen viele leere Buckets untersucht werden, bevor das nächste Element gefunden wird.  \cite{brown_cq}\cite{wuttisittikulkij_study_2003} \\
Brown schlägt zur Behebung letzteren Problems eine Strategie vor, nach der nach einem eventfreien Jahreszyklus das kleinste Element von allen Buckets zurückgegeben wird. \cite{brown_cq}
\subsection{Erweiterung: Dynamic Calendar Queue}
In der \ac{DCQ} wird die Heuristik zur Berechnung von \textit{\ac{bw}} beim Kopieren in eine größere bzw. kleinere Queue verändert. Es werden nicht die ersten \textit{n} Elemente zur Berechnung der durchschnittlichen Zeit zwischen den Events genutzt, sondern Elemente aus dem Bucket, in dem die meisten Events sind. So soll ein repräsentativeres Ergebnis der Zwischeneventzeit erzielt werden.
Außerdem werden zwei Metriken eingeführt: Die durchschnittlichen \textit{enqueue}- und \textit{dequeue}-Kosten ($C_E$ und $C_D$). Wenn diese einen vorher definierten Grenzwert überschreiten, wird eine Neuberechnung von \textit{\ac{bw}} initiiert, sodass sich die Queue an Änderungen in der Eventdistribution \textit{dynamisch} anpasst. \cite{dcq}\cite{snoopy}
